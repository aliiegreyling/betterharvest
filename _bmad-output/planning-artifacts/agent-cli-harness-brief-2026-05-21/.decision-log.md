# Decision Log: Agentic Workflow CLI Harness Brief

## 2026-05-21

- Created initial product brief for an Agentic Workflow CLI Harness.
- Positioned BMAD as the workflow spine and Serena MCP as the semantic code intelligence layer.
- Defined MCP-native extensibility as a foundational requirement.
- Scoped the initial version around orchestration, context awareness, model switching, tool discovery, and scaffold planning.
- Deferred implementation stack, provider selection, approval model, and deployment target decisions to PRD/architecture.
- Pulled and analyzed current Forge v0.1 implementation as the concrete starting point.
- Decided the product brief should evolve from abstract harness concept to Forge-based harness roadmap.
- Recorded current implementation baseline: TypeScript CLI, Anthropic-only model routing, SDLC phase runner, local tools, run state, audit and cost views.
- Identified missing gaps: MCP registry, Serena integration, BMAD artifact wiring, brownfield maintenance, context broker, provider-agnostic model switching, approvals, and deployment planning.
- Treated TypeScript/Node, CLI-only operation, local filesystem run state, and Forge as accepted baseline decisions for the next PRD pass.
- Implemented first v0.2 foundation slice in Forge: project context detection, MCP registry/health, Serena config awareness, BMAD artifact writing, model override flags, context-budget flags, status/context/design/work commands, and README updates.
- Verified with `npm run typecheck`, `npm run build`, and built CLI smoke tests for `status`, `mcp list`, `mcp health`, `models`, `context refresh`, `design`, and `work`.
- Regenerated BMAD test artifacts with portable project-relative context and verified no local paths or secret patterns are present.
- Decided Forge should be chat-first: running `forge` opens an interactive model chat, while slash commands invoke operational project journeys.
- Added `/request` as the low-token capture path and retained scriptable Commander subcommands for automation.
- Added a Forge diagnostics layer with user-friendly errors, next-step guidance, recoverable chat command failures, and opt-in verbose debugging via `FORGE_DEBUG=1`, `forge --verbose`, or `/set debug true`.
